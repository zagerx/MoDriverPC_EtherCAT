# 当前框架架构问题与边界收缩方案

> 本文基于当前最新源码分析，描述当前存在的架构边界问题，并提出边界收缩方案。

---

## 一、当前存在的问题

### 问题 1：`EcatController` 职责过多，违反单一职责原则

`EcatController` 当前承担的职责已收敛为：

1. **状态迁移**：`TransitionTo()` / `DoStepTo()` / `DoInit()` / `DoScan()` / `DoEnterMaintenance()` / `DoShutdown()` 等
2. **节点管理代理**：`GetSlaveCount()` / `ForEachSlaveNode()`

**已剥离职责**：
- **Activity 调度** → `ActivityScheduler`
- **周期任务** → `ProcessDataEngine`

**后果**：
- 状态迁移逻辑与 Activity 执行、PDO 周期等运行期逻辑已解耦。
- `GetSlaveNodeManager()` / `GetEcMaster()` 仍保留为 `ProcessDataEngine` 等底层周期模块专用接口，但不再对 `EcatApplication` 暴露，避免上层越界。

---

### 问题 2：`EcatApplication` 直接访问 `EcatController` 的内部依赖

当前 `EcatApplication::OnParam()` 和 `ExecuteActivityForAllNodes()` 通过 `controller_.GetSlaveNodeManager()` 拿到 `SlaveNodeManager` 后遍历节点：

```cpp
auto &node_manager = controller_.GetSlaveNodeManager();
for (size_t i = 0; i < node_manager.GetNodeCount(); ++i) {
    SlaveNode *node = node_manager.GetNode(i);
    // ...
}
```

**后果**：
- `EcatApplication` 越过了 `EcatController` 的封装边界，直接操作从站节点管理器。
- 从站遍历逻辑散落在应用层，节点管理器的内部结构变更会影响 `EcatApplication`。
- 职责边界模糊：`EcatController` 到底负责什么？`EcatApplication` 到底负责什么？

---

### 问题 3：Activity 调度与状态机耦合

当前 `ExecuteActivity()` 实现在 `EcatController` 内部：

```cpp
bool EcatController::ExecuteActivity(std::unique_ptr<EcatActivity> activity)
{
    // 1. 检查 state_ 是否允许启动
    // 2. 执行 activity->Execute()
    // 3. 根据失败策略调用 EnterErrorState() / Stop()
}
```

**后果**：
- Activity 的调度、失败处理与状态机迁移逻辑耦合。
- Activity 执行期间同步阻塞，状态机无法响应其他命令或执行周期任务。
- 如果未来 Activity 变长（如 FOE 升级），这种设计会失效。

---

### 问题 4：周期任务归属不清

`RunOneCycle()` 和 `CheckSlaveStates()` 已移入 `ProcessDataEngine`，由 `EcatApplication::Run()` 调用。周期任务归属已清晰，不再与状态迁移耦合。

当前仍存在的问题：`src/cyclic/cyclic_runner.h/.cpp` 已经实现但完全未被使用，是一个死模块，计划在阶段 4 处理。

---

### 问题 5：模块间调用关系不清晰

当前调用关系：

```text
main.cpp
    └── EcatApplication::Run()
            ├── CommandReader::Read()
            ├── EcatController::GetState()
            ├── EcatController::Scan() / EnterMaintenance() / StartOperation() / Stop()
            ├── EcatController::RunOneCycle() / CheckSlaveStates()
            ├── EcatController::ExecuteActivity()
            └── EcatController::GetSlaveNodeManager()  ← 越界
```

`EcatApplication` 既做命令分发，又做周期任务触发，又做 Activity 编排，还直接遍历节点。`EcatController` 成了大杂烩。

---

## 二、重构目标

通过**边界收缩**，让每个模块只负责一件事，模块间只通过最小、稳定的接口通信。

目标状态：

```text
main.cpp
    └── EcatApplication::Run()          // 状态机单步 + 命令分发
            ├── CommandReader::Read()   // 命令输入
            ├── EcatController          // 状态迁移 + 状态查询
            ├── ActivityScheduler       // Activity 调度
            └── ProcessDataEngine       // PDO 周期 + 状态监控
```

---

## 三、具体方案

### 3.1 重新定义 `EcatController`：只做状态迁移和状态保护

**保留职责**：
- 维护 `ControllerState`
- 执行状态迁移：`Initialize()` / `InitializeAdapter()` / `Scan()` / `EnterMaintenance()` / `StartOperation()` / `Stop()`
- 状态查询：`GetState()` / `GetSlaveCount()` / `StateToString()` / `IsInitialized()` / `IsOperational()`
- 错误状态进入：`EnterErrorState()`（仅被 `ActivityScheduler` 和 `ProcessDataEngine` 调用）

**删除职责**：
- 删除 `ExecuteActivity()` / `IsActivityRunning()`
- 删除 `RunOneCycle()` / `CheckSlaveStates()`（移到 `ProcessDataEngine`）

**受限保留**：
- `GetEcMaster()` / `GetSlaveNodeManager()` 保留为 `ProcessDataEngine` 等底层周期模块专用接口，不对外层应用暴露。

**新增接口**：
- `ForEachSlaveNode(std::function<void(SlaveNode &)> callback)`：安全遍历节点，不暴露 `SlaveNodeManager`

```cpp
class EcatController {
public:
    // 生命周期
    bool Initialize(const EcMasterConfig &config);
    bool InitializeAdapter(const EcMasterConfig &config);
    bool Scan();
    bool EnterMaintenance();
    bool StartOperation();
    void Stop();

    // 状态查询
    ControllerState GetState() const;
    size_t GetSlaveCount() const;
    static const char *StateToString(ControllerState state);
    bool IsInitialized() const;
    bool IsOperational() const;

    // 安全遍历节点，不暴露 SlaveNodeManager
    void ForEachSlaveNode(const std::function<void(SlaveNode &)> &callback);

private:
    bool TransitionTo(ControllerState target);
    bool DoStepTo(ControllerState next);
    bool DoStartOperation();
    bool DoInit();
    bool DoScan();
    bool DoEnterMaintenance();
    bool DoPdoConfigure();
    bool DoSafeOp();
    bool DoDcConfigure();
    bool DoOperational();
    bool DoShutdown();
    void EnterErrorState(const std::string &reason);

    ControllerState state_ = ControllerState::kUninitialized;
    EcMaster master_;
    SlaveNodeManager node_manager_;
    EcMasterConfig config_;
    std::vector<SlaveInfo> slave_infos_;
};
```

---

### 3.2 新增 `ActivityScheduler`：独立 Activity 调度

`ActivityScheduler` 只负责 Activity 的启动检查、同步执行和失败处理。

```cpp
class ActivityScheduler {
public:
    explicit ActivityScheduler(EcatController &controller);

    // 执行 Activity，返回是否成功
    bool Execute(std::unique_ptr<EcatActivity> activity);

    // 是否有 Activity 正在执行
    bool IsRunning() const;

private:
    EcatController &controller_;
    std::atomic<bool> running_{false};
};
```

**职责边界**：
- 读取 `controller_.GetState()` 检查 `CanStart()`
- 执行 `activity->Execute()`
- 根据 `GetFailurePolicy()` 调用 `controller_.EnterErrorState()` 或 `controller_.Stop()`，或保持状态
- 不直接访问任何从站、节点或 `EcMaster`

---

### 3.3 新增 `ProcessDataEngine`：独立周期任务引擎

`ProcessDataEngine` 只负责 PDO 周期收发和从站状态监控。

```cpp
class ProcessDataEngine {
public:
    explicit ProcessDataEngine(EcatController &controller,
                               EcMaster &master,
                               SlaveNodeManager &node_manager);

    // 执行一次 PDO 周期（仅在 kOperational 有效）
    void RunOnce();

    // 检查从站状态（在 kMaintenance / kOperational 有效）
    void CheckSlaveStates();

private:
    EcatController &controller_;
    EcMaster &master_;
    SlaveNodeManager &node_manager_;
};
```

**职责边界**：
- 读取 `controller_.GetState()` 决定是否执行任务
- 发现异常时调用 `controller_.EnterErrorState()`
- 不发起任何状态迁移

---

### 3.4 重新定义 `EcatApplication`：只做编排

`EcatApplication` 是应用层的"胶水"，负责：

1. 单步执行状态机（`Run()`）
2. 非阻塞读取命令
3. 根据当前状态分发命令
4. 调用 `EcatController` 执行状态迁移
5. 调用 `ActivityScheduler` 执行 Activity
6. 调用 `ProcessDataEngine` 执行周期任务

**持有成员**：

```cpp
class EcatApplication {
public:
    EcatApplication(std::unique_ptr<CommandReader> command_reader,
                    EcatController &controller,
                    ActivityScheduler &scheduler,
                    ProcessDataEngine &engine);

    bool Initialize(const EcMasterConfig &config);
    bool Run();
    void Shutdown();

private:
    std::unique_ptr<CommandReader> command_reader_;
    EcatController &controller_;
    ActivityScheduler &scheduler_;
    ProcessDataEngine &engine_;
};
```

**禁止做的事**：
- 不直接访问 `SlaveNodeManager`
- 不直接调用 `TransitionTo()` / `DoXxx()` / `EnterErrorState()`
- 不直接执行 Activity
- 不直接操作 `EcMaster`

**遍历节点的正确方式**：

```cpp
void EcatApplication::OnDiagnose()
{
    LOG_INFO << "Running SDO diagnostics...";
    controller_.ForEachSlaveNode([this](SlaveNode &node) {
        scheduler_.Execute(std::make_unique<SdoDiagnosticsActivity>(node));
    });
}
```

---

### 3.5 重新定义 `main.cpp`：只做生命周期和调度

```cpp
int main(int argc, char *argv[])
{
    // 1. 配置
    EcMasterConfig config;
    // ...

    // 2. 创建核心对象
    EcatController controller;
    ActivityScheduler scheduler(controller);
    ProcessDataEngine engine(controller, controller.GetEcMaster(), controller.GetSlaveNodeManager());

    auto app = std::make_unique<EcatApplication>(
        std::make_unique<StdinCommandReader>(),
        controller,
        scheduler,
        engine);

    if (!app->Initialize(config)) {
        return 1;
    }

    // 3. 注册信号
    RegisterShutdownSignals();

    // 4. 周期调度
    while (g_running.load()) {
        if (!app->Run()) {
            break;
        }
        std::this_thread::sleep_for(std::chrono::milliseconds(1));
    }

    // 5. 清理
    app->Shutdown();
    return 0;
}
```

> 注：如果 `ProcessDataEngine` 需要与 `EcatApplication::Run()` 分离调度，也可以由 `main` 直接周期调用 `engine.RunOnce()` 和 `engine.CheckSlaveStates()`，而不在 `Run()` 内部调用。

## 四、模块边界规则

### `EcatApplication` 允许调用

- `EcatController::Initialize()` / `InitializeAdapter()` / `Scan()` / `EnterMaintenance()` / `StartOperation()` / `Stop()`
- `EcatController::GetState()` / `GetSlaveCount()` / `StateToString()`
- `EcatController::ForEachSlaveNode()`
- `ActivityScheduler::Execute()` / `IsRunning()`
- `ProcessDataEngine::RunOnce()` / `CheckSlaveStates()`
- `CommandReader::Read()`

### `EcatApplication` 禁止调用

- `EcatController::TransitionTo()` / `DoXxx()` / `EnterErrorState()`
- `EcatController::GetSlaveNodeManager()`（删除）
- `EcatController::ExecuteActivity()`（删除）
- 任何 `EcMaster` / `SlaveNodeManager` 接口

### `ActivityScheduler` 允许调用

- `EcatController::GetState()`
- `EcatController::EnterErrorState()`
- `EcatController::Stop()`
- `EcatActivity::CanStart()` / `Execute()` / `GetFailurePolicy()`

### `ActivityScheduler` 禁止调用

- `EcatController::TransitionTo()` / `DoXxx()`
- 任何 `EcMaster` / `SlaveNodeManager` / `SlaveNode` 接口（除非通过 Activity）

### `ProcessDataEngine` 允许调用

- `EcatController::GetState()`
- `EcatController::EnterErrorState()`
- `EcMaster::RunOneCycle()` / `CheckAllSlavesInState()` 等周期相关接口
- `SlaveNodeManager::UpdateAllOutputs()` / `UpdateAllInputs()`

### `ProcessDataEngine` 禁止调用

- `EcatController::TransitionTo()` / `DoXxx()` / `Stop()`
- 任何状态迁移接口

## 六、实施步骤建议

### 阶段 1：Activity 调度剥离（低风险）

1. 新增 `ActivityScheduler`。
2. 把 `EcatController::ExecuteActivity()` / `IsActivityRunning()` 逻辑搬到 `ActivityScheduler`。
3. `EcatController` 删除这两个方法。
4. `EcatApplication` 改为持有 `ActivityScheduler` 引用，调用 `scheduler_.Execute()`。
5. 编译测试。

### 阶段 2：节点访问收敛（低风险）

1. `EcatController` 新增 `ForEachSlaveNode()`。
2. `EcatApplication` 中所有 `GetSlaveNodeManager()` 调用改为 `ForEachSlaveNode()`。
3. 删除 `EcatController::GetSlaveNodeManager()` 公共接口。
4. 编译测试。

### 阶段 3：周期任务剥离（中风险）✅ 已完成

1. 新增 `ProcessDataEngine`。
2. 把 `EcatController::RunOneCycle()` / `CheckSlaveStates()` 逻辑搬到 `ProcessDataEngine`。
3. `EcatController` 删除这两个方法。
4. `EcatApplication` 改为持有 `ProcessDataEngine` 引用，在 `Run()` 中调用。
5. `main.cpp` 创建并注入 `ProcessDataEngine`。
6. 编译测试通过。

### 阶段 4：`CyclicRunner` 处理（当前待处理）

- 方案 A：删除 `CyclicRunner`，因为当前 `main` 的 `while + sleep` 已足够。
- 方案 B：把 `CyclicRunner` 改造为真正的周期调度器，替代 `main` 中的手写循环。

---

## 七、预期收益

1. **`EcatController` 职责单一**：只做状态迁移，不再做 Activity 调度和周期任务。
2. **`EcatApplication` 不越界**：不再直接访问 `SlaveNodeManager`。
3. **Activity 调度独立**：后续可以自然扩展为异步执行、取消、优先级等。
4. **周期任务归属清晰**：`ProcessDataEngine` 明确负责 PDO 和状态监控。
5. **模块边界可测试**：每个模块可以独立单元测试。
6. **`CyclicRunner` 有归宿**：死模块被删除或激活。

---

## 八、不做的范围

本次边界收缩方案**不解决以下问题**，留待后续：

1. **Activity 异步化**：当前仍保持同步执行，未来在 `ActivityScheduler` 内部扩展。
2. **多线程并发**：当前仍是单线程模型，`ProcessDataEngine` 与 `EcatApplication` 串行执行。
3. **配置文件加载**：仍由 `main.cpp` 硬编码 `EcMasterConfig`。
4. **单站 Activity 指定**：当前 Activity 仍针对所有从站，后续可扩展命令参数。
